# Internationalization (i18n)

HARI ships in **English and French**. Every user-facing string is translatable, and
the AI assistant answers in the active language.

## Approach: `next-intl`, cookie-based (no i18n routing)

Locale is a **per-user preference**, not part of the URL. There are no `/en` or `/fr`
path prefixes and **no middleware** — the locale lives in a `NEXT_LOCALE` cookie. This
fits an authenticated internal tool and composes cleanly with the existing server-side
auth (`requireUser()`), which stays exactly as it was.

See the official docs: [next-intl without i18n routing](https://next-intl.dev/docs/getting-started/app-router/without-i18n-routing).

## Where things live

| Path | Role |
|---|---|
| `src/i18n/routing.ts` | `locales` (`['en','fr']`) + `defaultLocale` (`en`). |
| `src/i18n/request.ts` | Per-request config — reads the cookie, loads the matching catalog. Wired up by the next-intl plugin in `next.config.ts`. |
| `src/i18n/locale.ts` | `getUserLocale()` / `setUserLocale()` server actions over the `NEXT_LOCALE` cookie. |
| `messages/en.json`, `messages/fr.json` | The translation catalogs. **`en.json` is the source of truth** for the typed keys (`global.d.ts`). |
| `src/components/layout/language-switcher.tsx` | FR/EN dropdown (in the sidebar and on the login page). Writes the cookie via `setUserLocale`; the server action's completion re-renders the tree. |
| `src/app/layout.tsx` | Sets `<html lang>` from the locale and wraps the app in `NextIntlClientProvider`; metadata via `getTranslations`. |

Keys are organized into namespaces: `metadata`, `common`, `nav`, `roles`,
`permissions`, `leaveType`, `leaveStatus`, `dashboard`, `directory`, `timeOff`,
`settings`, `chat`, `reasoning`, `citations`, `leave`, `directoryCard`, `payslip`,
`tools`, `login`, `languageSwitcher`.

## Using translations

- **Server Components** (default): `const t = await getTranslations("namespace")`.
- **Client Components** (`"use client"`): `const t = useTranslations("namespace")`.
- **Dates / numbers**: use `useFormatter()` / `getFormatter()`, or for currency the
  `formatMAD` helper in `src/lib/utils.ts` — never hand-roll `toLocaleString`.
- Keys are **type-checked** against `en.json` (see `global.d.ts`), so a typo or a
  missing key is a compile error, and `npx tsc --noEmit` catches drift.

Pluralization and interpolation use ICU syntax in the catalog, e.g.
`"{count, plural, one {# day} other {# days}}"` and
`"{status, select, APPROVED {approved} other {updated}}"`.

## The AI assistant

`src/app/api/chat/route.ts` reads the locale and instructs the model to **respond in
the user's language** (and to relay any tool result, including errors, in that
language). "Today" in the system prompt is formatted for the locale. Tool *result*
strings stay English at the data layer (they're unit-tested verbatim in
`tests/tools.integration.test.ts`); the model translates them when it relays them to
the user. Model-facing tool `description`s are intentionally left in English.

## Adding to it

- **A new string**: add the key to **both** `messages/en.json` and `messages/fr.json`
  (same path), then reference it with `t("…")`. `tsc` will flag a key present in one
  catalog but missing in the other only once it's used; keep the two in lockstep.
- **A new locale**: add it to `locales` in `src/i18n/routing.ts` and create
  `messages/<locale>.json` with the full key set.

## Known boundary

Currency is currently fixed to **MAD** (`formatMAD`) and timestamps are **UTC**. Making
both configurable by a super admin (and making the assistant aware of them) is tracked
as a follow-up — see the org-settings work, not part of this i18n change.
